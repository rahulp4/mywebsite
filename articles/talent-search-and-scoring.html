How a résumé becomes a score.

A single opening can draw hundreds of résumés. Behind a "best match: 87%" badge sits a real pipeline: an AI reads each PDF into structured data, a job description becomes a weighted checklist, and every candidate is compared and scored against it — explainably. Let's build that pipeline from the ground up, with something to poke at each step.

Start with the pipeline ↓

01 — The shape of the problem

Two halves: parse, then match

Every résumé arrives as a document — a PDF or Word file built for human eyes, not for comparison. And every job is a wishlist in a recruiter's head. To rank one against the other, a machine has to do two very different jobs, in order.

First, parse. Turn each messy résumé into the same tidy structured profile — name, experience, skills, dates — so two candidates can be compared field by field. Then, match. Turn the job description into a rubric and score each profile against it. This article walks the whole path; the demo below is the map.

▶ Step through the pipeline

One résumé's journey from uploaded file to a saved score. Press Next to advance a stage.

Notice the split. The parse half is one POST /api/v1/parsewithllm call. The match half is a second service that takes two JSON blobs — the job's rubric and the candidate's profile — and returns a score. The rest of the article is those two halves, up close.

02 — The parse endpoint

Turning a résumé into data

The parsing endpoint is deliberately tiny on the outside: you POST a .pdf or .docx file, and it returns structured JSON. Inside, it runs four steps — and the interesting one is that a large language model does the actual reading.

the four steps
  1. Validate — reject anything that isn't a .pdf or .docx.
  2. Extract text — pull the raw words out of the document.
  3. LLM parse — hand that text to Gemini with a schema, and get back structured fields (experience, skills, education, dates).
  4. Derive metrics — compute things the LLM shouldn't guess at: total experience, tenure, job switches (that's the next section).
📇 An analogy

A résumé is a hand-written index card in someone's own style — abbreviations, gaps, a personal layout. The LLM is a patient assistant who reads any card and copies it onto one standard form. Once every candidate is on the same form, they line up in the same boxes and you can finally compare them.

Why a language model rather than rules? Because résumés don't follow rules. "Sr. SWE II", "Senior Software Engineer", and "Software Eng. (L5)" all mean the same job; dates appear as "2019–Present", "Jan 2019", or "03/19". A model reads intent; a regex reads only what you anticipated.

    03 — Derived signals
    
Doing math on time

    
Some facts you never trust an LLM to compute — you compute them yourself, deterministically, from the dates it extracted. "How many years of experience?" sounds trivial until you realise people hold overlapping jobs: a full-time role and a side contract, or two titles at one company. Add the durations naïvely and you'll credit someone with 15 years of experience in a 10-year career.

    
The fix is interval merging: lay every job on a timeline, fuse the overlaps into single continuous stretches, and measure the union. Drag the jobs below and watch the naïve sum diverge from the honest total.


    

      ▶ Overlapping jobs, one honest total
      
      

        

        

        

        

      

      

        —naïve sum (double-counts)
        —merged union (the truth)
        —org switches
      

    


    

      ∑ The formal bit
      
Total experience is the length of the union of the job intervals, in years; job switches count distinct employers, minus the one you're currently in:

      
years = |⋃ intervals| / 365.25 · switches = max(0, |companies| − 1)

      
The same union trick powers "years with Kubernetes" and "skills used in the last 6 years" — every duration in the profile is a merged interval, never a naïve sum. Deterministic, auditable, and immune to the LLM's arithmetic.

      

        [sᵢ,eᵢ] one job's start → end
        ∪ union — overlaps merged, counted once
        365.25 days/year (leap-adjusted)
      

    

  







  

    04 — The job description as data
    
A rubric, not a paragraph

    
Now the other half. A job description in prose can't be scored — "strong Java, cloud-native, 10+ years" is for humans. So the system turns it into a rubric: a list of requirements, each one a small, explicit rule.

    
Every rule answers four questions:

    

      

        data — what to look for ("Java", "≥10")
        matchreq — how to compare (next section)
        weightage — how much it matters (1–5)
        profiledatasource — where to look in the profile
      

    

    
That last one is subtle and powerful. A single requirement can point at several places in the candidate's profile, and the matcher takes the best score across them. "Technical skills" looks in languages, frameworks, tools, platforms, and the technologies listed on each job — so a skill only mentioned inside a project description still counts.


    

      ▶ One requirement, as JSON
      
A real rule from the sample job description. Every requirement on the rubric has this exact shape.

      

      
weightage: 5 is a must-have; sourcecondition: "OR" means matching any listed value (any acceptable location) is enough.

    


    
A recruiter never writes this JSON by hand — a wizard does. Each requirement is a known field with a sensible default weight and matcher; the recruiter fills in the values and nudges the weights. The result is a complete, machine-scorable rubric.

  







  

    05 — The methodology
    
Three ways to compare (and a fourth for meaning)

    
Not every requirement is compared the same way — that's the core idea. A list of skills, a paragraph about the role, and a "10+ years" bar are different kinds of thing, and each needs its own kind of match. The rubric names the method per field with matchreq. Try each below.

      ∑ The formal bit
      
Jaccard — overlap of two sets of tags. Cosine — the angle between two texts turned into vectors. Operator — a hard numeric gate, pass or fail.

      
J = |A∩B|/|A∪B| · cos = u·v/(|u||v|) · s = 1[x ⋈ t]

      
Cross-encoder has no tidy formula here — it's a small neural model that reads both strings together and rates their similarity, which is how it knows Kubernetes ≈ GKE and Golang ≈ Go even though they share no letters. That semantic reach is why skills use it instead of plain Jaccard.

      

        A,B two sets of tags
        u,v texts as number vectors
        ⋈ a comparator: ≥, >, <
        1[·] 1 if true, else 0
      

    

  







  

    06 — From many scores to one
    
Rolling it up into a single number

    
Each requirement now yields a 0–100 score. The last step blends them into one headline number — but not a plain average, because a must-have and a nice-to-have shouldn't count the same. Each score is multiplied by its weight, and the whole thing is normalised so it stays on a 0–100 scale.


    

      ▶ The score breakdown — tune it live
      
Real requirements from the sample job description. Drag each weight and score and watch the weighted total move. Notice how a low score on a heavy row hurts far more than on a light one.

      

      

        —overall (weighted)
        —plain average, for contrast
      

    


    

      ∑ The formal bit
      
overall = Σ(sᵢ·wᵢ) / Σ(wᵢ)

      
A hard gate makes this vivid: "years of experience" is an operator at weight 5. Miss the bar and its score is 0, which drags 0×5 through the sum and pulls the whole number down hard — exactly the behaviour you want for a non-negotiable.

      
And every row is auditable: the match service returns, per requirement, which candidate value it scored and the runners-up it considered — so "why 72%?" always has a paper trail.

      

        sᵢ requirement i's score (0–100)
        wᵢ its weight (1–5)
      

    

  







  

    07 — Wrap-up
    
Recap, use cases & pitfalls

    
End to end: a résumé is parsed into a standard profile, a job becomes a weighted rubric, each requirement is matched with the right method, and the pieces roll up into one explainable score that ranks the pile.


    
The ideas, in one place

    

      

        parse PDF/DOCX → LLM → structured profile
        derive durations via merged intervals
        rubric data · matchreq · weight · source
        jaccard set overlap for tags
        vector cosine for meaning
        operator hard numeric gate
        cross-encoder semantic skill match
        roll-up Σ s·w / Σ w
      

    


    
Where it earns its keep

    
Ranking the pile. Hundreds of applicants become a sorted, filterable list — "show me everyone above 75%" — instead of a folder nobody opens. Explainable screening. Because each score carries its evidence, a recruiter (or an audit) can see why, not just what. Consistent rubrics. The same weighted criteria apply to every candidate, which is both faster and fairer than mood-of-the-afternoon screening.


    
Pitfalls worth knowing

    
Garbage in. If the parse misreads a scanned PDF, every downstream score inherits the error — extraction quality is load-bearing.

    
Proxy signals. Some fields are inferred, not stated — "work mode" is guessed from where past jobs were located, so it carries a deliberately low weight. Know which signals are real and which are stand-ins.

    
Weight tuning is the product. The matchers are just math; the judgement lives in the weights. Set them badly and a great candidate ranks below a keyword-stuffer.

    
Wrong matcher, wrong answer. Score skills with plain set-overlap and "GKE" won't match "Kubernetes". The semantic matcher exists precisely so equivalent tech isn't missed — using the wrong one silently drops good people.

    
Match well, and the machine doesn't replace the recruiter — it hands them a ranked, reasoned shortlist and gives back the hours they'd have spent getting there.
