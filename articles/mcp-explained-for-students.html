MCP Explained the Way You'd Actually Ask About It

A student-friendly walkthrough of Model Context Protocol — what problem it solves, how it differs from plain function calling, and how the client-server split actually works.

July 12, 2026 · ~10 min read

If you've been building anything with Claude or GPT lately, you've probably heard the term MCP thrown around — Model Context Protocol. Most explanations jump straight into architecture diagrams and protocol specs. This post takes the slower, more honest route: the exact questions a curious developer would ask, in the order they'd ask them, until the picture actually clicks.

01 — The problem

First, what problem are we even solving?

Imagine you're building an AI app — say, a portfolio tracker that uses Claude to answer "how diversified is my portfolio?" To answer that, Claude can't just think its way to your holdings. It needs to actually fetch data from somewhere — your broker's API, in this case.

That's the core problem: an LLM is great at reasoning and language, but it can't reach outside itself. Every time it needs real data or needs to do something in the real world, some piece of code has to bridge that gap.

Here's the mess that shows up once you have more than one AI app and more than one external service, and what changes once MCP sits in the middle:

Before MCP: every app writes custom code for every tool. After MCP: apps and tools connect through one shared protocol.

Two apps, three tools, six completely separate pieces of integration code. Add a third app or a fourth tool and the number explodes. This is called the N × M problem — N apps, each independently solving the same M integration problems, over and over.

Once Gmail, Slack, and GitHub each expose themselves through MCP once, any app that speaks MCP — Claude, an IDE agent, anything — can use all of them without writing tool-specific code.

02 — The common confusion

"Wait, isn't this what function calling already does?"

This is the question that trips most people up, and it's a fair one — because at the surface level, the code you write for a function-calling tool does end up making a remote API call, just like MCP eventually does too.

Here's the distinction. Function calling (OpenAI's "function calling" and Anthropic's "tool use" are the same underlying idea) solves exactly one problem: how does the model tell your app what it wants to do? You describe a function as a JSON schema, the model decides to call it, and returns a structured request instead of guessing in plain text.

{
  "name": "fetch_unread_emails",
  "description": "Get unread emails from the user's Gmail inbox",
  "input_schema": {
    "type": "object",
    "properties": {
      "max_results": { "type": "integer" }
    }
  }
}

But function calling says nothing about who writes the code that actually executes that function. That part is still on you. In a typical Java app, that looks like:

class GmailIntegration {
    OAuth2Credentials creds;

    List<Email> fetchUnread() {
        String token = creds.refreshAccessToken();
        HttpResponse resp = httpClient.get(
            "https://gmail.googleapis.com/gmail/v1/users/me/messages?q=is:unread",
            Map.of("Authorization", "Bearer " + token)
        );
        return parseGmailJson(resp.body());
    }
}

Auth handling, endpoint shape, response parsing — all hand-written, all specific to Gmail. Build a second app that also needs Gmail access, and someone writes this exact class again, probably slightly differently, probably with slightly different bugs.

So the honest answer is: function calling and MCP aren't rivals, they solve two different halves of the same journey.

  • Model ↔ App: "The model wants to call fetch_unread_emails." → solved by function/tool calling.
  • App ↔ External service: "Someone has to actually talk to Gmail's API." → this is what kept getting rewritten per app, and this is what MCP standardizes.
Function calling connects the LLM to your app. MCP connects your app to the real service. Two different layers, not competitors.
03 — Who talks to whom

So who talks to whom — client, server, and the LLM

Once that separation is clear, the natural next question is: does the MCP client decide which tool gets called?

No — and this is the one place it's easy to get backwards. The LLM decides. The MCP client just carries out that decision.

MCP Server (owned by the service — Gmail, Slack, a stock broker): exposes what tools exist and their schemas, contains the real implementation (auth, API calls, parsing), and knows nothing about the LLM or the conversation.

MCP Client (lives inside your app — Claude, an IDE agent): asks the server "what tools do you have?", hands those schemas to the LLM, waits for the LLM to decide which tool to call and with what arguments, forwards that decision to the server, and returns the result back to the LLM.

A single turn of conversation, step by step:

The LLM decides what to call. The MCP client carries out the decision. The MCP server does the actual work.

In code, an MCP client sitting inside a Java app looks like this:

class McpClientApp {
    AnthropicClient llm;         // calls to Claude's API
    McpConnection brokerServer;  // connection to the broker's MCP server

    void handleUserMessage(String userInput) {
        List<Tool> tools = brokerServer.listTools(); // ask the server what it can do

        LlmResponse response = llm.sendMessage(userInput, tools);

        if (response.wantsToolCall()) {
            String toolName = response.getToolName();      // e.g. "get_portfolio_holdings"
            Map<String, Object> args = response.getToolArgs();

            ToolResult result = brokerServer.callTool(toolName, args);

            String finalAnswer = llm.sendToolResult(result);
        }
    }
}

Notice what this class does not contain: no broker-specific auth code, no hand-parsed JSON shapes. That's living inside the broker's MCP server instead — written once, by people who understand that API best, and reusable by every MCP-speaking app that wants portfolio data.

04 — The plain-language version

The version to explain to a non-technical friend

Strip away the code and protocol names, and here's the plain-language version worth keeping in your back pocket:

Think of an AI assistant as a smart new employee. Brilliant at thinking and talking, but to actually get things done — check email, look up a bank balance — they need access to other people's systems.

Before MCP, every company hiring an AI employee had to individually train them on every system: "here's how you log into Gmail," "here's the secret handshake for the bank." Ten companies, ten AI employees, all learning the same handshakes independently — a lot of duplicated effort, and more room for mistakes.

MCP is every service publishing an official instruction manual once, in a common language, so any AI employee can read it and know how to work with that system from day one.

One nuance worth keeping even in this simple version: the AI still does the thinking — deciding when to check the inbox. MCP doesn't replace that judgment. It just guarantees that once the decision is made, there's a standard, reliable way to carry it out.

05 — Wrap-up

The takeaway

MCP isn't a replacement for function calling — it's the missing other half. Function calling standardized how a model expresses intent. MCP standardizes how that intent gets executed against the real world, so that the same integration — Gmail, Slack, a stock broker's API — doesn't need to be rebuilt from scratch inside every AI app that wants to use it.
